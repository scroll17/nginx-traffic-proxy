# nginx-traffic-proxy — Обзор платформы

> Единая защищённая точка входа для множества веб-приложений на одном сервере:
> маршрутизация по доменам, автоматический TLS, бесшумные обновления без простоя и
> закрытый доступ к базам данных по mTLS.

Документ для ознакомления (команда / заказчик). Технические детали реализации и
внутренний аудит — в отдельных документах.

---

## 1. Что это

Платформа-прокси, через которую публикуются несколько независимых приложений
(каждое — со своим доменом) с одного хоста. Обеспечивает:

- 🌐 **Один вход на все домены** — маршрутизация HTTPS-трафика по доменному имени.
- 🔒 **Автоматический TLS** — выпуск и продление сертификатов Let's Encrypt на каждое приложение.
- ♻️ **Бесшовные обновления** — обновление proxy-слоя без простоя сервиса (blue-green).
- 🛡️ **Закрытый доступ к БД** — базы недоступны из интернета; подключение по поддомену только с клиентским сертификатом (mTLS).
- 🚧 **Защита периметра** — лимиты запросов/соединений, anti-slowloris, изоляция сетей, сокрытие версий.

## 2. Архитектура

```
                                  Internet
                                     │
                          :80   :443   :7443
                                     │
                    ┌────────────────▼─────────────────┐
                    │          HAProxy (edge)           │   L4 / TCP passthrough
                    │   blue-green · real-IP (PROXY v2) │
                    └────────────────┬──────────────────┘
                ┌────────────────────┴────────────────────┐
        ┌───────▼────────┐                       ┌──────────▼───────┐
        │   nginx_blue   │   ← катятся по        │   nginx_green    │
        │  (front nginx) │     очереди, 0 down   │  (front nginx)   │
        └───────┬────────┘                       └─────────┬────────┘
                └─────────────────────┬─────────────────────┘
                              маршрутизация по домену
        ┌──────────────┬──────────────┴───────────┬───────────────────┐
   ┌────▼────┐    ┌────▼────┐               ┌──────▼──────┐
   │  SWAG   │    │  SWAG   │   …           │ DB-шлюзы    │  (mTLS)
   │ (app A) │    │ (app B) │               │ (ghostunnel)│
   └────┬────┘    └────┬────┘               └──────┬──────┘
        │ TLS терм.    │                           │ plaintext (внутри)
   client/api      client/api               postgres / mongo / redis
```

**Три слоя:**

| Слой | Роль |
|---|---|
| **Edge (HAProxy)** | Принимает трафик из интернета, балансирует на backend-nginx, переживает их обновление без простоя |
| **Front (nginx ×2)** | Маршрутизация по домену, автоматический TLS (ACME), форс HTTPS, защита от флуда |
| **Apps (SWAG + сервисы)** | Терминация TLS, само приложение, база данных, mTLS-шлюз к БД |

## 3. Сетевая топология и подсети

Сети изолированы по уровням — каждый «прыжок» в своей сети, приложения не видят
балансировщик напрямую.

```
┌─ edge-network ── 10.123.0.0/24 ───────────────────────────────────────┐
│   haproxy (nginx_lb)                                                   │
│   nginx_blue ─┐   nginx_green ─┐         (фронт состоит в двух сетях)   │
└───────────────┼────────────────┼──────────────────────────────────────┘
                │                │
┌─ nginx-proxy-network ── (bridge) ─────────────────────────────────────┐
│   nginx_blue ─┘   nginx_green ─┘                                       │
│   synergate_swag   juscov_swag   …   *_pg_gateway  *_mongo_gateway …   │
└───────────────┬───────────────────────────────────────────────────────┘
                │  (каждое приложение — в своей внутренней сети)
┌─ <app>-internal ── (bridge, без выхода наружу) ───────────────────────┐
│   client    api    postgres    mongo    redis                         │
└───────────────────────────────────────────────────────────────────────┘
```

| Сеть | Подсеть | Кто в ней | Назначение |
|---|---|---|---|
| `edge-network` | `10.123.0.0/24` (закреплена) | haproxy, nginx_blue/green | Edge ↔ front; основа anti-spoofing real-IP |
| `nginx-proxy-network` | назначается Docker | nginx_blue/green, все swag/шлюзы | Front ↔ приложения |
| `<app>-internal` | назначается Docker | client, api, БД приложения | Внутренние сервисы приложения (без выхода в интернет) |

## 4. Карта портов

| Порт | Контейнер | Протокол | Назначение | Доступен из |
|---|---|---|---|---|
| **80** | HAProxy | TCP/HTTP | ACME-challenge + редирект на HTTPS | 🌍 Интернет |
| **443** | HAProxy | TCP/TLS (passthrough) | HTTPS-трафик приложений | 🌍 Интернет |
| **7443** | HAProxy | TCP/TLS (mTLS) | Доступ к БД по поддомену | 🌍 Интернет (только с client-cert) |
| 8404 | HAProxy | HTTP | Liveness для healthcheck | 🔒 внутр. |
| 81 | nginx_blue/green | HTTP | Health-проба для HAProxy | 🔒 edge |
| 80 / 443 / 7443 | nginx_blue/green | TCP | Приём от HAProxy (PROXY v2) | 🔒 edge |
| 443 | SWAG | TLS | Приём от front (passthrough) | 🔒 proxy-net |
| 80 | SWAG | HTTP | ACME / редирект | 🔒 proxy-net |
| 8443 | DB-шлюзы (ghostunnel) | TLS (mTLS) | Приём от front | 🔒 proxy-net |
| 5432 / 27017 / 6379 | postgres / mongo / redis | TCP | Только внутри приложения | 🔒 app-internal |

🌍 = опубликовано на хост · 🔒 = доступно только внутри соответствующей docker-сети.
**Порты БД (5432/27017/6379) наружу не публикуются вообще.**

## 5. Потоки трафика

**Веб (HTTPS):**
```
client ─TLS─▶ HAProxy:443 ─▶ nginx (роутинг по SNI/домену) ─▶ SWAG приложения (терминирует TLS) ─▶ app
```
Реальный IP клиента сохраняется на всём пути.

**Веб (HTTP :80):** только ACME-проверка сертификата и `301 → https`. По обычному HTTP приложение не отдаётся.

**Доступ к БД (:7443):**
```
psql / mongosh / redis-cli ─▶ локальный туннель (client-cert) ─mTLS─▶ HAProxy:7443
                           ─▶ nginx (роутинг по поддомену) ─▶ mTLS-шлюз ─▶ база данных
```

## 6. Что обеспечивает безопасность

- 🔐 **Шифрование в транзите** — весь внешний трафик по TLS (Let's Encrypt).
- 🚪 **Базы данных закрыты** — порты не опубликованы; доступ только по поддомену с клиентским сертификатом (mTLS) и паролем.
- 🧱 **Сегментация сети** — балансировщик отделён от приложений; приложения — во внутренних сетях без прямого выхода наружу.
- 🛡️ **Защита от перегрузки** — лимиты запросов и соединений на IP, тайм-ауты против slowloris.
- 🕵️ **Анти-спуфинг** — реальный IP клиента нельзя подделать с уровня приложений.
- 🙈 **Сокрытие инфраструктуры** — версии скрыты; неизвестные домены отбрасываются без ответа.

## 7. Технологический стек

| Технология | Назначение |
|---|---|
| HAProxy 3.0 | Edge-балансировщик (L4), blue-green |
| nginx 1.27 (stream + http) | Маршрутизация по домену, ACME, лимиты |
| LinuxServer SWAG | TLS-терминация + Let's Encrypt на приложение |
| ghostunnel | mTLS-шлюзы к базам данных |
| PostgreSQL / MongoDB / Redis | Базы данных приложений |
| Docker / Compose | Оркестрация и сетевая изоляция |

## 8. Эксплуатация

| Действие | Команда | Простой |
|---|---|---|
| Поднять платформу | `docker compose up -d` | — |
| Добавить/убрать домен | правка `nginx/domains/*.map` → `make domains` | 0 |
| Обновить proxy-слой | `make roll` | 0 (на уровне сервиса) |
| Проверить конфигурацию | `make validate` | — |
| Доступ к БД | `cd app/db-gateway/<db> && ./tunnel.sh` | — |
